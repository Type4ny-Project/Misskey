## 重要ルール: question ツールの使用

**全てのエージェント・サブエージェントは、ユーザーへの質問を行う際に必ず `question` ツール（questiontools）を使用してください。**

- テキストメッセージで質問を投げかけることは禁止です。
- 選択肢を提示する場合、確認を求める場合、曖昧な指示を明確化する場合、全て `question` ツールを使ってください。
- これはプロジェクト全体の統一ルールです。

## 設計書エージェント

汎用の設計書オーケストレーターと、Misskey 専用の分析サブエージェントが用意されています。

### 使い方

**Tab キーで `planner` に切り替えて、「○○の設計書を作って」と言うだけ。**

```
マルチテナントの設計書を作って
OAuth2 対応の設計書を作って
Push 通知リファクタの設計書を作って
```

planner はトピックに応じて自律的に:
1. 必要な分析タスクを決定
2. 分析エージェントを並列起動（db-analyzer, api-analyzer, impact-analyzer 等）
3. 結果を統合して `{トピック名}.md` を生成

### エージェント一覧

| エージェント | 種別 | 役割 |
|---|---|---|
| `planner` | **Primary** | 汎用設計書オーケストレーター。トピックに応じた分析→設計書生成を自律実行する。 |
| `db-analyzer` | hidden subagent | DB 構造分析（TypeORM エンティティ、マイグレーション、クエリ） |
| `api-analyzer` | hidden subagent | API 分析（エンドポイント、認証、ストリーミング、ActivityPub） |
| `impact-analyzer` | hidden subagent | 影響範囲調査（バックエンド・フロントエンド横断） |
| `reviewer` | hidden subagent | コードレビュー（セキュリティ、パフォーマンス、Misskey 規約） |

### スキル一覧

| スキル名 | 役割 |
|---|---|
| `mt-analysis` | マルチテナント分析の共通コンテキスト・ガイドライン |
| `mt-migration-plan` | マイグレーション計画テンプレート |

### マルチテナント方針

- **分離方式**: 既存の `host` カラム / `Host` ヘッダーをそのまま活用（新規 `tenantId` ではない）
- **テナント識別**: サブドメインまたはカスタムドメイン
- **核心の課題**: `host == null` → ローカル判定の全面差し替え → `isManagedHost(host)` 判定層の導入
- 連合とローカルテナントは同じ `host` 概念で扱う
