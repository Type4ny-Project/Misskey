## マルチテナント方針

当面のMVPでは、同一DBに統合する方式ではなく、`Host` ごとに別々のDBへ接続を振り分ける方式を優先する。

- テナント解決は HTTP `Host` ヘッダーで行う
- 各 tenant は独立した PostgreSQL DB を持つ
- アプリケーションプロセスは共有してよい
- Misskey 本流との差分は、できるだけ `tenant runtime` / 接続ルーティング層に閉じ込める
- 既存コード全体に `host == null` の意味変更を広げる方針は避ける

## 実装時の優先順位

- まず `Host` ごとの `config` / `db` / `meta` / queue 文脈を正しく切り替える
- そのうえで、非同期ジョブやキャッシュが別 tenant に漏れないことを確認する
- 同一DB論理分離のための大規模な条件分岐追加は、MVPでは行わない

## テストゴール

最終的なゴールは、以下が確認できること。

- 適当なサーバー2つを tenant として起動できる
- `Host` によって別DBへ正しく接続される
- tenant A/B でそれぞれローカルユーザー作成・投稿ができる
- tenant A/B 間で連合が成立する
- 実際に ActivityPub の配送・受信まで通る

## レビュー観点

マルチテナント関連の変更では、特に以下を重点的に確認すること。

- request 外の処理でも tenant 文脈が失われていないか
- queue job data に tenant 識別子が載っているか
- cache / redis / pubsub / object storage が tenant をまたいで衝突しないか
- 本流マージ時に広範囲衝突しない構造になっているか
